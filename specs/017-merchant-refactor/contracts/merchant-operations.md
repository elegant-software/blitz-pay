# Contract: Merchant Operations

## Purpose

Define the externally visible merchant and branch contract changes required by the operational merchant refactor.

## Merchant Read Model

Every merchant response exposes:

- `merchantId`
- `merchantCode`
- `merchantName`
- `merchantStatus`
- `businessVertical`
- `contactInfo`
  - `website`
  - `email`
  - `phoneNumber`
- `address`
  - `addressLine1`
  - `addressLine2`
  - `city`
  - `postalCode`
  - `country`
- `location`
  - `latitude`
  - `longitude`
  - `geofenceRadiusMeters`
  - `googlePlaceId`
  - optional place enrichment fields
- `offerings`
  - list of enabled merchant offering codes

## Merchant Create/Update Rules

- Merchant create/update requests use operational merchant language only.
- Onboarding workflow fields and onboarding status transitions are not part of the operational merchant contract.
- Merchant name is required.
- Merchant contact, address, and location blocks are optional.

## Branch Read Model

Every branch response exposes:

- `branchId`
- `merchantId`
- `branchCode`
- `branchName`
- `branchStatus`
- `contactOverride`
  - `website`
  - `email`
  - `phoneNumber`
- `address`
  - `addressLine1`
  - `addressLine2`
  - `city`
  - `postalCode`
  - `country`
- `location`
  - `latitude`
  - `longitude`
  - `geofenceRadiusMeters`
  - `googlePlaceId`
  - optional place enrichment fields

## Branch Create/Update Rules

- Branch create/update requests require:
  - `branchName`
  - `branchStatus`
  - structured address
  - geospatial location including geofence radius
- Branch website, email, and phone overrides are optional.
- `branchCode` is generated by the system and never supplied by callers.

## Merchant Offering Rules

- Merchant offerings come from a fixed platform-managed catalogue.
- Merchant contracts expose enabled offerings as merchant-scoped values only.
- Branch contracts do not expose per-branch offering overrides.
- `DEFERRED_PAYMENT` cannot be enabled unless `PRE_ORDER` or `WALK_IN_ORDERING` is also enabled.
- `APPOINTMENT_BOOKING` may appear as an enabled offering even though appointment workflow remains out of scope in this refactor.
